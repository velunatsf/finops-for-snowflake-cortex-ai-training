A

New SERVICE_TYPE Billing

Breaking Change for Cost Dashboards

Four services are getting their own billing types as part of the AI Credits transition. If your cost dashboards only look for AI_SERVICES, you will silently miss spend from these new service types.

Moving to Own SERVICE_TYPE

  • CORTEX_AGENTS
  • CORTEX_CODE_CLI
  • CORTEX_CODE_SNOWSIGHT
  • SNOWFLAKE_INTELLIGENCE

Still Under AI_SERVICES

  • AI Functions (COMPLETE, SENTIMENT, etc.)
  • Cortex Analyst
  • Fine Tuning
  • Search Serving
  • Provisioned Throughput

Query: Detect New AI Credit Service Types

Transition Check

When to run: Immediately - check if your account already shows the new service types in METERING_DAILY_HISTORY.

SQL 
SELECT
    SERVICE_TYPE,
    MIN(USAGE_DATE)          AS first_seen,
    MAX(USAGE_DATE)          AS last_seen,
    SUM(CREDITS_BILLED)      AS total_credits_billed,
    SUM(CREDITS_USED)         AS total_credits_used
FROM SNOWFLAKE.ACCOUNT_USAGE.METERING_DAILY_HISTORY
WHERE SERVICE_TYPE IN (
    'AI_SERVICES',
    'CORTEX_AGENTS',
    'CORTEX_CODE_CLI',
    'CORTEX_CODE_SNOWSIGHT',
    'SNOWFLAKE_INTELLIGENCE'
)
GROUP BY SERVICE_TYPE
ORDER BY total_credits_billed DESC;
Action Required for Your Dashboards

If you built cost dashboards that filter on SERVICE_TYPE = 'AI_SERVICES', update them now to also include the four new service types. Otherwise, your AI spend totals will be silently wrong - showing lower costs than actual.

B

Spend Cap - Cortex Code Credit Guardrails

Spend Caps on Budgets

Enable financial control by setting automated spend limits on your cloud expenses. Uses out-of-the-box budgets to enforce automated cost boundaries at the project level for specific services in a non-destructive manner. On budget breach, usage is blocked and financial damage is controlled - reduce bill shock, and protect your Cortex budget!

How do you govern LLM spend from Cortex Code on a per-user basis? Three tools, each for a different scenario:

Layer Tool Behavior Scope
Proactive Per-user credit limits Hard block - rolling 24h window Account-wide default + per-user override
Threshold Monthly Budgets Alert before overshoot - no hard block Account-wide
Audit Usage History views Attribution after-the-fact by user + model Per surface (CLI / Snowsight)

Per-User Credit Limits Proactive - Blocks Before Spend Happens

Parameter Controls
CORTEX_CODE_CLI_DAILY_EST_CREDIT_LIMIT_PER_USER Cortex Code CLI usage
CORTEX_CODE_SNOWSIGHT_DAILY_EST_CREDIT_LIMIT_PER_USER Cortex Code in Snowsight usage

Set Account-Wide Default + Per-User Override

Guardrail SQL
SQL 
-- Account-wide default: 20 credits/day per user (CLI)
ALTER ACCOUNT SET CORTEX_CODE_CLI_DAILY_EST_CREDIT_LIMIT_PER_USER = 20;

-- Tighter limit for a specific user
ALTER USER USER_COCO SET CORTEX_CODE_CLI_DAILY_EST_CREDIT_LIMIT_PER_USER = 5;

-- Same for Snowsight
ALTER ACCOUNT SET CORTEX_CODE_SNOWSIGHT_DAILY_EST_CREDIT_LIMIT_PER_USER = 20;
ALTER USER USER_COCO SET CORTEX_CODE_SNOWSIGHT_DAILY_EST_CREDIT_LIMIT_PER_USER = 5;

Rolling 24h Window

The limit is a rolling 24-hour window, not a calendar day. If a user hits the cap at 9am Tuesday, they're blocked until 9am Wednesday.

Independent Surfaces

CLI and Snowsight are tracked independently with separate parameters. Cap someone on CLI while leaving Snowsight open, or vice versa.

Two-Level Control

Set an account-wide default, then override per user. Power users get more headroom; restricted roles get tighter limits.

Monthly Budgets Threshold - Alerts Before Overshoot

Snowflake Budgets track actual billed credits across warehouses, pipelines, and AI together. Set a monthly limit and get notified before you hit it.

Budgets Alert - They Don't Hard-Block

Budgets are an early warning system. Per-user credit limits are the only hard stop for Cortex Code spend.

Cortex Code Usage History Audit - Attribution After-the-Fact

CORTEX_CODE_CLI_USAGE_HISTORY

Breaks down Cortex Code CLI spend by user and model. One row per usage event. Use this to identify who's spending the most and on which models.

CORTEX_CODE_SNOWSIGHT_USAGE_HISTORY

Same structure for Snowsight-based Cortex Code usage. CLI and Snowsight tracked in separate views - query both for complete attribution.

Query: Top Cortex Code Spenders by User & Model

Attribution
SQL 
-- Cortex Code CLI spend by user and model (last 30 days)
SELECT
    h.USER_NAME,
    f.key                       AS model_name,
    SUM(h.TOKEN_CREDITS)        AS total_credits,
    ROUND(total_credits * 3, 4) AS est_cost_usd
FROM SNOWFLAKE.ACCOUNT_USAGE.CORTEX_CODE_CLI_USAGE_HISTORY h,
     LATERAL FLATTEN(input => h.TOKENS_GRANULAR) f
WHERE h.USAGE_TIME >= DATEADD('day', -30, CURRENT_TIMESTAMP())
GROUP BY h.USER_NAME, f.key
ORDER BY total_credits DESC;

-- Repeat with CORTEX_CODE_SNOWSIGHT_USAGE_HISTORY
-- for Snowsight usage attribution

Key Observations
  • TOKENS_GRANULAR breaks down input vs output tokens per model

Resource Monitors Do NOT Cover AI Token Costs

Resource Monitors only cap warehouse compute credits. If that's your current AI cost control strategy, you have a gap. Use per-user credit limits as the hard stop, Budgets for early warning, and the usage history views for attribution.
